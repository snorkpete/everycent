# System Audit ‚Äî 2026-04-02

Re-audit of the Vue 3 codebase. Evaluates current state after significant Phase 4 refactoring (EcFormDialog, EcPageLayout, EcItemList extraction; useAllocationGrouping, useCurrentAndPastBudgets, useWantsNeedsBudgetBreakdown composables; authTokens centralization; interceptor extraction; factory infrastructure; shared stubs).

## How to Read This

- **P0** ‚Äî Fix before adding new features. Bugs, broken contracts, or patterns that actively cause problems.
- **P1** ‚Äî Fix during this refactoring phase. Architectural debt that compounds with every new feature.
- **P2** ‚Äî Fix opportunistically. Real issues but low blast radius or only relevant at scale.
- **P3** ‚Äî Nice to have. Style consistency, minor DRY violations, cosmetic.

**DUP-CSS-01: `.toolbar-left` / `.toolbar-right` toolbar layout ‚Äî 6 files**

Near-identical CSS for page toolbars. All follow the same pattern: flex container with gap, alignment, and shrink behaviour.

| File | Lines | Exact Match? |

| `BudgetsPage.vue` | 202-212 | `.toolbar-left` + `.toolbar-right` |

All share this core pattern:
```css
.toolbar-left {
  display: flex;
  align-items: center;
  gap: 0.75rem;
  flex: 1;
  min-width: 0;
}
.toolbar-right {
  display: flex;
```

**Assessment:** This pattern belongs in `EcPageLayout`. The toolbar slot already exists ‚Äî EcPageLayout should own the flex container that splits toolbar content into left/right zones. ## 3. Store Health

### Overall Pattern Assessment

All stores now follow a consistent pattern:
- `loading`, `error` refs with proper `finally` blocks
- Error handling with `e instanceof Error ? e.message : fallback`
- API layer used consistently (no more direct `apiGateway` calls in stores)

**STOR-01 (missing finally blocks): RESOLVED** ‚Äî budgetListStore.ts now has `finally` blocks on all 5 async methods.

**STOR-02 (duplicated budget fetching): RESOLVED** ‚Äî `useCurrentAndPastBudgets` composable extracted. Used by transactionStore.ts:18 and importStore.ts:26.

**STOR-03 (settingsStore bypasses API): RESOLVED** ‚Äî settingsStore.ts:23 now calls `bankAccountApi.getOpen()`.

**STOR-04 (inconsistent edit mode): PARTIALLY RESOLVED** ‚Äî budgetStore, transactionStore, and sinkFundStore all follow the same pattern: `enterEditMode()`, `exitEditMode()`, `cancelEdit()` (refetch on cancel). The "drafts" pattern discussed in the previous audit was not adopted ‚Äî all stores use the simpler "refetch on cancel" approach, which is consistent.

**STOR-05 (settings always available): PARTIALLY ADDRESSED** ‚Äî BudgetPage.vue:83 calls `settingsStore.fetchAll()` alongside store.fetch() on mount. TransactionsPage.vue:143 calls `settingsStore.fetchAll()` on mount. However, there is no system-level router guard. Individual pages are responsible for fetching settings.

**STOR-06 (child components push to store arrays): PARTIALLY RESOLVED**
- `BudgetIncomeList.vue:101` now calls `store.addIncome(newIncome)` (store action added at budgetStore.ts:59-62)
- `BudgetAllocationList.vue:285` still directly pushes: `store.budget.allocations.push(newAllocation)` ‚Äî the store has `addAllocation()` but the component doesn't use it
- `SinkFundsPage.vue:121` directly pushes: `store.sinkFund.sink_fund_allocations.push(...)` ‚Äî no store action exists

### Remaining Store Issues

**STOR-NEW-01: BudgetAllocationList bypasses `store.addAllocation()`** (P3)
`BudgetAllocationList.vue:285` calls `store.budget.allocations.push(newAllocation)` directly despite `budgetStore.addAllocation()` existing at budgetStore.ts:63-66. Minor ‚Äî should use the action for consistency.

**STOR-NEW-02: SinkFundsPage directly mutates `store.sinkFund.sink_fund_allocations`** (P3)
`SinkFundsPage.vue:118-127` pushes to the array directly. Need a `store.addObligation()` action.

**STOR-NEW-03: Duplicated `isAutoAllocateEligible` logic** (P2)
`transactionStore.ts:140-146` and `importStore.ts:213-219` contain identical eligibility checks. Extract to a shared utility.

**STOR-NEW-04: `futureBudgetsStore` fetches its own settings independently** (P2)
`futureBudgetsStore.ts:69` calls `futureBudgetsApi.getSettings()` to get settings, while other stores use `useSettingsStore()`. This store has its own local `settings` ref that isn't connected to the central settingsStore. If/when settings become a system-level concern (STOR-05), this will need to be reconciled.
