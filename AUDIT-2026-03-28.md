## A. Rails Import Services

The import pipeline (preview + save) evolved independently, creating duplicated validation logic, inconsistent error handling, and scattered field definitions. These findings form a single coherent refactoring effort.

Existing test coverage is comprehensive (25+ specs per service), so this is safe to refactor with confidence.

### P0

**RAIL-01: Test-code mismatch on allocation_id**
`spec/services/import_save_service_spec.rb:540-549` expects `allocation_id` to be blocked during import, but `import_save_service.rb:5` later added it to `PERMITTED_FIELDS`. Test will fail.
*Fix:* Update test to match current intent (allocation_id IS allowed, for auto-allocate), or remove from PERMITTED_FIELDS if it shouldn't be allowed.

### P1

**RAIL-02: Shared validation/classification layer missing**
Preview and save services duplicate IBAN validation, date parsing, duplicate detection, and period checking. Both do the same work, but with different error handling (preview raises, save skips silently).

This is the core issue — RAIL-03 through RAIL-06 are symptoms of this.

*Fix:* Extract a `TransactionClassifier` that takes a transaction and returns its status (new, duplicate, out_of_period, invalid_date). Extract shared validation (IBAN, amounts) into a module. Both services call the classifier — preview uses the classifications to project balances, save uses them to decide what to persist. Single pass over transactions.

The error handling philosophy unifies naturally: classify first (never raise for individual transactions), then the service decides what to do with each classification. Service-level raises only for fatal issues (IBAN mismatch on the account itself).

**RAIL-03: Permitted fields defined in multiple places**
`import_save_service.rb:4-6` (PERMITTED_FIELDS), `transactions_controller.rb:89-114` (import_save_params, import_preview_params) — three separate field lists that can diverge.
*Fix:* Centralize to a single source of truth that both controller and service reference.

**RAIL-04: Response schema mismatch between preview and save**
Preview returns `{ net, projected_balance, transactions }`. Save returns `{ net: 0, saved_count, skipped }` with `net` hardcoded to 0. Frontend must handle two different shapes.
*Fix:* Unify response structure or make the difference explicit and documented. Should be addressed as part of the RAIL-02 refactor.
*Note:* This also affects the Vue frontend (importStore, ImportPage) — the FE code adapts to whatever schema the BE returns. Track FE changes as a follow-up when we get to Vue refactoring.

### P2

**RAIL-05: AutoAllocationSuggester — fragmented DB queries**
`auto_allocation_suggester.rb:40-52` loads sink fund IDs separately then filters. Could be a single query with a JOIN.
*Fix:* Refactor to `Transaction.joins(:bank_account).where(bank_account: { account_type: 'normal' })`.

**RAIL-06: Missing validation — both withdrawal and deposit non-zero**
Preview doesn't check that both amounts aren't simultaneously non-zero.
*Fix:* Add validation rule if this is a real business constraint. Decide during RAIL-02 refactor.

### P3

**RAIL-07: No format strategy pattern for imports**
Adding a new import format requires duplicating validation logic or adding conditionals.
*Fix:* Not needed now. The RAIL-02 refactor will naturally make this easier by separating classification from persistence. If a second server-side format is added, extract a format adapter at that point.

**RAIL-08: Foreign key constraints commented out**
`db/migrate/20180819114217_add_foreign_keys.rb:15-31` — all FKs disabled. Pre-existing, not new.
*Fix:* Before enabling, two checks are needed: (1) Query for orphaned references in production data — years of data may have integrity issues that have been silently accumulating. (2) Assess whether FKs would break multi-step operations that depend on insert order (e.g., creating transactions before allocations exist during import). Not a "just flip it on" change. Start with a data audit, then decide.

## B. Quality Gates & Enforcement

No enforcement mechanisms exist. Strong tooling (ESLint, Prettier, strict TypeScript) with zero automation. This section establishes the mechanical guardrails before refactoring begins.

### P1

**QUAL-01: Pre-commit hooks (husky + lint-staged)**
No husky, no lint-staged. Code that fails lint/type-check can be committed freely.

Husky is a Node package that installs git hooks into the repo via committed config files (`.husky/pre-commit`). Any developer who runs `npm install` gets the hooks automatically — no manual setup. lint-staged pairs with it to run linters only on staged files, keeping pre-commit fast even when the broader codebase has issues.

*Fix:* Install husky + lint-staged. Pre-commit runs ESLint + Prettier on staged files.

**QUAL-02: Fix production TypeScript errors (92 errors)**
546 total type-check errors. Split: 92 in production code, 454 in test files.

Production errors cluster in a few areas:
- Importers (~30) — `abnAmroBankImporter.ts`, `abnAmroCreditCard2026Importer.ts` — "possibly undefined" on array access
- `importStore.ts` (~12) — "possibly undefined" on account/transaction lookups
- `futureBudgetsStore` / `BudgetMassEditDialog` (~5) — "possibly undefined"
- `EcDateField` (3) — date callback type mismatch
- `SettingsPage` (3) — string | undefined mismatch
- `budgetListStore` (1), `AdjustBalancesDialog` (1), `BudgetAllocationList` (2)
- One `replaceAll` error — needs `lib: 'es2021'` in tsconfig

*Fix:* Fix the 92 production errors (single focused session per cluster). Gate production code immediately — pre-commit rejects new production TS errors. These should be proper fixes with guards, not bang assertions.

**QUAL-03: Test file TypeScript errors — incremental ratchet (454 errors)**
Test files have 454 type errors, mostly "Object is possibly undefined" and loose typing on mock responses. Fixing all at once would be mentally exhausting and isn't necessary.

*Fix:* Record 454 as the baseline count. Ratchet down as we touch each area during refactoring — when we modify a test file, fix its type errors too. Track progress to ensure the count only goes down, never up.

**QUAL-04: Clean up existing lint and formatting violations**
15 ESLint errors (unused vars, `any` in tests) plus any Prettier violations. These are the kind of issues that auto-fix can handle — `lint --fix` and `prettier --write`. Get to a clean baseline before the pre-commit hook enforces it going forward.
*Fix:* Run `npm run lint -- --fix` and `npm run format`, then manually fix anything that remains. Do alongside QUAL-02.

### P2

**QUAL-05: Make `build:safe` the default build command**
`npm run build` doesn't run type-check — only `build:check` does. Pre-commit should catch everything, but defense in depth is worth it.
*Fix:* Rename/restructure npm scripts: `build` (default) runs checks then builds. `build:force` skips checks for when speed is of the essence. Pre-commit is the primary gate; build checks are the safety net.

**QUAL-06: CI pipeline (GitHub Actions — free tier only)**
No GitHub Actions configured. Happy to use free minutes; not willing to pay for CI as a solo dev.
*Fix:* Set up GitHub Actions within free tier limits. Run type-check, lint, test, build on push/PR. Evaluate if free minutes are sufficient for the repo's push frequency. If not, pre-commit hooks remain the primary enforcement.

## D. API Gateway & Auth

### P1

**GATE-01: Extract modular interceptors**
Auth tokens, loading state, error handling, 401 redirect — all inline in one 66-line file. Loading callbacks are untestable inline lambdas (known TODO at `api-gateway.spec.ts:119-123`).

*Fix:* Extract interceptors into separate files (`src/api/interceptors/`). Each interceptor is an exported creator function that receives its dependencies. This makes them independently testable, resolves the spec TODO, and makes the gateway file a simple setup/wiring file.

**GATE-02: Auth token management scattered**
localStorage accessed directly in api-gateway (attach/save headers), authStore (clear credentials), handle401 (clear and redirect). Three independent actors manipulating the same storage.

*Fix:* Create a single `AuthTokenManager` (store or utility) that owns token read/write/clear. Gateway interceptors and authStore both use it. One place to change if storage mechanism changes.

**GATE-03: Dual logout paths**
`handle401()` in api-gateway and `authStore.logOut()` both clear credentials independently. Can diverge (e.g., handle401 clears tokens but authStore.loggedIn stays true).

*Fix:* handle401 emits an event (or calls a shared function), authStore handles the actual logout. One code path for credential cleanup.

**GATE-04: No error normalization**
`axios.isAxiosError()` leaks into domain code (authStore.ts:19). No standard error type — consumers must know about Axios.

*Fix:* Create an `ApiError` class. Response interceptor normalizes errors before they reach consumers. Domain code catches `ApiError`, not Axios types.

*Dropped:*
- ~~GATE-05: Tight coupling to Axios~~ — The gateway IS the abstraction. The one place that leaks Axios (`authStore.ts` using `axios.isAxiosError()`) gets fixed by GATE-04 (error normalization).
- ~~GATE-06: No token refresh logic~~ — Not necessary. Session expiry on a personal finance app just means logging in again. Not worth the complexity.

## E. Vue Stores & API Layer

### Design Context

Everycent is an SPA technically, but from the user's perspective each page is a standalone app. There's no expectation of state persistence across page navigations — switching pages resets state. The only persistence between refreshes is through URL query params (so refreshing keeps your place but clears WIP).

This means "stale state across stores" is less of a problem than it first appears — pages fetch fresh data on mount. The real issues are: duplicated fetch logic, API layer bypass, and unstructured mutations.

### P0

**STOR-01: budgetListStore — missing `finally` blocks**
`budgetListStore.ts:40-82` — five async methods don't reset `loading.value` on error. Loading stays true forever if an error is thrown.
*Fix:* Add `finally { loading.value = false }` to all async methods.

### P1

**STOR-02: Duplicated data-fetching logic**
`budgetsForDropdown` computed (sort open budgets, pick earliest as current) duplicated in transactionStore and importStore. `bankAccounts` fetched independently in 4 stores.

*Fix:* Part of this task is deciding the right approach: shared utility functions (`getCurrentBudget(budgets)`), a composable, or just standardizing the pattern across stores. Given the page-per-visit model, utility functions are likely simplest — each store still fetches its own data, but the logic for processing it is shared. Decide during implementation.

**STOR-03: settingsStore bypasses API layer**
`settingsStore.ts:24` directly calls `apiGateway.get('/bank_accounts')` instead of using `bankAccountApi`.
*Fix:* Use `bankAccountApi.getOpen()`. The comment about "avoiding coupling to bankAccountStore's loading state" is a valid concern, but the fix is using the API function, not the store.

**STOR-04: Inconsistent edit mode pattern**
transactionStore uses drafts (clone on edit, discard on cancel). budgetStore mutates live state (refetch on cancel).
*Fix:* Standardize on one pattern. Drafts are cleaner (no wasted refetch on cancel). Consider a `useEditableResource()` composable, but only if it doesn't add complexity — the pattern may be simple enough to just follow consistently without an abstraction.

**STOR-05: Settings should always be available on page load**
`transactionStore.ts:150-156` and `importStore.ts:223-229` read `settingsStore.settings.primary_budget_account_id` to decide if auto-allocate should be offered. If settings haven't been fetched, this silently returns wrong results.
Settings are used in multiple places across the app. Rather than documenting per-store dependencies, make it a system-level responsibility: settings are always loaded on every page/route navigation. This is the one piece of state that can reasonably survive across pages, but we shouldn't depend on the cached version — just load fresh on every route change.
*Fix:* Add a router guard (or equivalent) that fetches settings before each page renders. Individual stores can then read settingsStore with confidence, no guards needed.

**STOR-06: Child components directly mutate store arrays**
`BudgetIncomeList.vue:101-102` pushes to `store.budget.incomes`. `BudgetAllocationList.vue:291` pushes to allocations. No validation, no events.
*Fix:* Add store actions (`store.addIncome()`, `store.addAllocation()`) that components call. Store can validate or enforce rules.

### P2

**STOR-07: budgetApi.getCurrentBudgetId() — dead code**
`budgetApi.ts:20` — defined but never called.
*Fix:* Remove.

**STOR-08: fetchList vs fetchAll naming**
`sinkFundStore.ts:43` uses `fetchList` while all other stores use `fetchAll`.
*Fix:* Rename for consistency.

### P3

**STOR-09: bankAccountStore.fetchInstitutions() — no loading state**
`bankAccountStore.ts:26-32` doesn't manage loading state.
*Fix:* Add loading state management.

**STOR-10: budgetListStore — catch(e) without type narrowing**
Should be `catch (e: unknown)` with `e instanceof Error` guard.
*Fix:* Align with reference pattern. Fix alongside STOR-01.

*Dropped:*
- ~~STOR-11: Fetch-after-mutation pattern~~ — Not a problem. Single Postgres instance, no read replicas. Refetch after save always returns consistent data. Extra round trip is negligible at current data volumes.

## F. Components & Reuse

### Design Principle: Build a Reusable Component Library

The same way `EcMoneyField` wraps all money display logic, we should systematically identify and extract helper components and composables throughout the system. The signals for extraction are:

1. **Repeated CSS** tied to a specific UI pattern across multiple components/pages
2. **Repeated logic** (computed properties, state management, formatting) across multiple components/pages
3. **Logic that doesn't belong** — business logic or formatting mixed into a component whose core responsibility is something else

Each signal points to a missing helper. The extraction builds up a library of reusable components and composables, with display consistency as an important byproduct.

Two systematic tasks support this:
- **COMP-00a: Identify extraction opportunities** — Audit all pages/components for repeated CSS, repeated logic, and misplaced logic. Output: a list of candidates with the files they'd replace.
- **COMP-00b: Extract and standardize** — For each candidate, build the shared component/composable (with config props where needed), replace all usages, remove inline code. Incremental — do as we touch each area.

The individual findings below are known candidates from this audit. More will surface during COMP-00a.

### P1

**COMP-01: BudgetAllocationList — 515-line responsibility blob**
Blends rendering, category grouping, allocation CRUD, calculations, dialog integration, hardcoded allocation classes. Multiple concerns that don't belong together.
*Fix:* Extract `useAllocationGrouping` composable for category data transformation. Move allocation classes to shared constants. Consider `AllocationRow.vue` for row rendering if it reduces complexity — but don't force it if the extraction doesn't clearly simplify things.

**COMP-02: Dialog footer pattern duplicated across 4+ dialogs**
BankAccountEditDialog, AllocationCategoryEditDialog, InstitutionEditDialog, SpecialEventForm — near-identical edit/view toggle footers (logic + CSS).
*Fix:* Extract `BaseEditDialog` component with slots for form content.

**COMP-03: Hardcoded colors instead of PrimeVue design tokens**
`#16a34a`, `#dc2626`, etc. in BudgetAllocationList, SinkFundAllocationTable, BudgetSummaryStrip, BudgetSummary. Inconsistent hex values for same semantic meaning.
*Fix:* Replace with PrimeVue CSS variables (`var(--p-green-600)`, `var(--p-red-600)`, etc.). Shared components should use tokens by default so consumers get consistency for free.

**COMP-04: EcMoneyField needs `inline` prop**
6+ components use `:deep()` overrides to make EcMoneyField work in tables (hide label, row layout). Classic case of repeated CSS pointing to a missing component feature.
*Fix:* Add `inline` prop to EcMoneyField that applies the table layout internally. Eliminates the `:deep()` workarounds.

**COMP-05: Budget breakdown calculations duplicated**
BudgetSummary.vue and BudgetSummaryStrip.vue duplicate needs/wants/savings percentage calculations. Repeated logic that doesn't belong in either component.
*Fix:* Extract `useBudgetBreakdown(budget)` composable.

**COMP-06: List page boilerplate duplicated**
BankAccountsPage, AllocationCategoriesPage, InstitutionsPage — identical dialog state management, edit/add handlers, save handling. Repeated logic across pages.
*Fix:* Extract `useListPage<T>()` composable.

### P2

**COMP-07: Table icon button — repeated CSS and interaction pattern**
`.eye-btn` styles (opacity transition on hover, icon styling) repeated in BudgetAllocationList, SinkFundAllocationTable, BudgetIncomeList, etc. Same CSS + same interaction logic in 5 files.
*Fix:* Extract `TableIconButton.vue` component.

**COMP-08: Implicit store-fetch contracts**
TransactionSearchForm assumes `store.fetchMetadata()` was called before mount. Silently renders empty if not.
*Fix:* Make data dependencies explicit via props, or document the initialization contract. Partially addressed by STOR-05 (settings always loaded), but applies more broadly.

**COMP-09: title vs v-tooltip inconsistency**
Mix of native `title` and PrimeVue `v-tooltip` across components.
*Fix:* Standardize on `v-tooltip`. Update CLAUDE.md (line 37 still references `title`).

### P3

**COMP-10: Dialog width units mixed (px vs rem)**
AllocationTransactionsDialog uses `500px`, everything else uses `rem`.
*Fix:* Standardize to `rem`. If BaseEditDialog (COMP-02) is extracted, enforce rem there.

**COMP-11: Section title + deleted row CSS duplicated**
Same `.section-title` and `.deleted-row` / `.closed-row` styles in 3+ files each. Repeated CSS pointing to missing shared utility classes.
*Fix:* Extract to shared CSS utility classes or small components.

**COMP-12: Z-index values inconsistent (5 vs 10 vs 15)**
Works correctly — category headers intentionally higher than regular headers. Just undocumented.
*Fix:* Define z-index scale as CSS custom properties if it becomes confusing. Low priority.

**COMP-13: No PrimeVue wrapper layer**
Raw PrimeVue components used directly everywhere. Design changes or PrimeVue major upgrades touch every page.
*Fix:* Consider only if a PrimeVue upgrade actually causes pain. Not worth the abstraction cost preemptively.

## G. Cross-Cutting Themes

### Theme 1: Rails Import Foundation (Section A)
RAIL-02 through RAIL-04 are symptoms of the same root cause: preview and save evolved without shared infrastructure. One coherent refactor (TransactionClassifier + shared validation + unified response approach) resolves them all. Self-contained, already has comprehensive tests. Do first.

### Theme 2: Quality Gates (Section B)
Prevent regressions mechanically. Pre-commit hooks (husky + lint-staged) are the highest-leverage single item — ESLint, Prettier, and type-check on staged files make simple violations impossible to commit. Fix the 92 production TS errors and 15 ESLint errors to establish a clean baseline, then gate immediately. Test file TS errors (454) ratchet down incrementally. `build:safe` becomes the default build command (checks then builds), with `build:force` as the escape hatch. CI via GitHub Actions free tier when ready.

### Theme 3: Test Confidence (Section C)
TEST-01 + TEST-02 + TEST-03 raise test confidence from 65% to 85%+. The key insight: mock at API boundaries only, use real stores. Shared factories and stubs reduce maintenance burden. Gradual migration — new tests use the new pattern, existing tests migrate as their source files are touched. Update testing patterns guide once the new patterns are established.

### Theme 4: Vue Code Quality (Sections D, E, F)
Three sub-themes:

**Gateway & Auth (Section D):** Extract modular interceptors, centralize auth token management, normalize errors. Makes the HTTP layer testable and the auth flow predictable.

**Stores (Section E):** Extract shared data-fetching utilities (not domain stores — the page-per-visit model means each page fetches fresh). Fix API layer bypass. Standardize edit mode pattern. Make settings a system-level concern loaded on every route change. Move array mutations into store actions.

**Components (Section F):** Systematically build a reusable component library. Identify repeated CSS, repeated logic, and misplaced logic across components — each is an extraction candidate. Known candidates include BaseEditDialog, EcMoneyField inline prop, budget breakdown composable, list page composable, table icon buttons. New candidates will surface during the systematic audit (COMP-00a).

Done last because quality gates and test infrastructure make this work safer.
